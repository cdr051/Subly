# 구독 관리 앱 — 기획 개요서 (PRD)

> 버전 0.1 · 구상 단계 · 작성일 2026-06-07
> 이 문서는 앱을 만드는 동안 "왜 / 무엇을 / 어떤 순서로" 판단하는 기준이 됩니다. 기능이 떠오를 때마다 여기에 적고, 우선순위가 바뀌면 이 문서를 먼저 고치세요.

---

## 1. 한 줄 정의

넷플릭스·유튜브 프리미엄 같은 **구독형 서비스를 한곳에서 관리**하고, 사용자의 성향과 사용 패턴을 바탕으로 **AI가 구독을 추천하거나 해지를 제안**하는 iOS 앱.

## 2. 해결하려는 문제

- 사람들은 자기가 매달 얼마를 어떤 구독에 쓰는지 정확히 모른다.
- 안 쓰는 구독을 잊고 계속 결제한다 (구독 누수).
- 새 서비스를 고를 때 비교 기준이 없다.
- 결제일/무료체험 종료일을 놓쳐 원치 않는 과금이 발생한다.

## 3. 타깃 사용자

- 구독 서비스를 3개 이상 쓰는 20~40대
- 지출을 관리하고 싶지만 가계부 앱은 번거롭다고 느끼는 사람
- "이거 아직도 결제되고 있었어?"를 경험해본 사람

## 4. 핵심 가치 제안

1. **한눈에 보기** — 모든 구독과 월/연 총지출을 한 화면에서
2. **놓치지 않기** — 결제일·무료체험 종료 전 알림
3. **똑똑하게 정리** — AI가 "이건 거의 안 쓰니 해지 고려", "이 취향이면 이 서비스가 더 맞을 수도"를 제안

## 5. 기능 범위 (단계별)

기능을 한 번에 다 만들려 하지 말고, 아래 순서대로 쌓습니다. **MVP를 먼저 완성해서 시뮬레이터에서 도는 것**이 1차 목표입니다.

### MVP (v1.0) — AI 없이 먼저 동작하게
가장 먼저 만들 핵심. AI는 일부러 뒤로 미룹니다. 데이터 구조와 화면이 안정돼야 AI를 얹을 수 있어요.

- 구독 추가 / 수정 / 삭제 (이름, 금액, 결제주기, 결제일, 카테고리)
- 구독 목록 화면 (한눈에 보기)
- 월간 / 연간 총지출 자동 계산 및 표시
- 결제일 알림 (로컬 알림)
- 데이터 로컬 저장 (SwiftData)

### v1.1 — 분석/시각화
- 카테고리별 지출 비중 (차트)
- 월별 지출 추이
- "사용 빈도" 수동 입력 (자주 씀 / 가끔 / 거의 안 씀)

### v1.2 — AI 기능 (이 앱의 차별점)
- **해지 제안**: 사용 빈도 낮음 + 비용 높음 구독을 AI가 정리 후보로 제시하고 이유를 설명
- **취향 기반 추천**: 사용자가 입력한 관심사/현재 구독을 바탕으로 대안·신규 서비스 제안
- **지출 요약 코멘트**: "이번 달 엔터테인먼트 비중이 높아요" 같은 자연어 인사이트

### v2.0 이후 — 본격 확장 (지금은 신경 끄기)
- iCloud 동기화 (여러 기기)
- 가족 공유 구독 관리
- 위젯 / 잠금화면
- 실제 결제 내역 연동 (이건 난도 높고 규제 이슈 있음 — 한참 뒤)

## 6. AI 기능 설계 (핵심 차별점이므로 별도 정리)

### 무엇을 AI에게 맡기나
AI는 "판단과 설명"을 담당합니다. 단순 합산·필터는 일반 코드로 처리하고, 사람 말투의 추천·해지 사유 생성·취향 매칭처럼 **규칙으로 짜기 어려운 부분**만 AI에 맡기세요.

### 어떻게 구현하나 (선택지)
- **방식 A — 클라우드 LLM 호출 (권장 시작점)**: 사용자의 구독 목록·사용빈도를 요약해 LLM API에 보내고, 추천/해지 사유를 자연어로 받아 화면에 표시. 구현이 가장 빠르고 품질이 좋음. 단, API 키 관리와 비용·프라이버시 고려 필요.
- **방식 B — 온디바이스 모델**: Apple의 온디바이스 프레임워크나 Core ML 모델 사용. 프라이버시·오프라인 강점이 있으나 구현 난도와 추천 품질 한계.
- **방식 C — 규칙 기반 "유사 AI"**: 점수 공식(비용 × (1/사용빈도))으로 해지 후보를 정렬하는 식. AI는 아니지만 MVP+ 단계에서 빠르게 "스마트한 느낌"을 줄 수 있음. 나중에 진짜 AI로 교체.

> 권장 경로: **C로 뼈대를 잡고 → A로 품질을 올린다.** 처음부터 LLM에 의존하면 비용·키관리·네트워크 처리까지 한꺼번에 떠안게 되어 진도가 안 나갑니다.

### 프라이버시 원칙 (처음부터 정해두기)
- 사용자 데이터는 기본 로컬 저장. AI 호출 시 **필요한 최소 정보만** 전송.
- "AI 추천을 위해 데이터를 사용한다"는 점을 사용자에게 명확히 고지하고 동의받기.
- App Store 심사 시 개인정보 처리방침이 필요하므로, 데이터 흐름을 지금부터 문서화.

## 7. 기술 스택

| 영역 | 선택 | 비고 |
|------|------|------|
| 언어 | Swift | |
| UI | SwiftUI | 선언형, 신규 프로젝트 표준 |
| 로컬 저장 | SwiftData | 모델에 `@Model`, 뷰에서 `@Query` |
| 작은 설정값 | `@AppStorage` | 테마, 통화 단위 등 |
| 알림 | UserNotifications | 결제일 로컬 알림 |
| 차트 | Swift Charts | Apple 기본 프레임워크 |
| 테스트(로직) | Swift Testing | `@Test` / `#expect` |
| 테스트(UI) | XCUITest | 필요 시에만 |
| AI | (1단계) 규칙 기반 → (2단계) LLM API | 위 6번 참고 |
| 동기화 | CloudKit | v2.0 이후 |

## 8. 데이터 모델 초안 (SwiftData)

실제 코드는 아니고, 어떤 데이터를 다룰지 잡아두는 스케치입니다.

**Subscription (구독)**
- `name`: 서비스 이름 (예: Netflix)
- `price`: 금액
- `currency`: 통화 (KRW, USD…)
- `billingCycle`: 결제주기 (월/연/주)
- `nextBillingDate`: 다음 결제일
- `category`: 카테고리 (엔터테인먼트, 생산성, 음악…)
- `usageFrequency`: 사용 빈도 (자주/가끔/거의 안 씀)
- `isTrial`: 무료체험 여부
- `trialEndDate`: 체험 종료일
- `notes`: 메모
- `iconName` 또는 `colorHex`: 시각적 구분용

**UserPreference (취향 — AI 추천용)**
- `interests`: 관심사 태그 목록
- `monthlyBudget`: 월 예산(선택)

## 9. 화면 구성 초안

1. **홈 / 대시보드** — 월·연 총지출, 다가오는 결제, 구독 카드 목록
2. **구독 상세** — 개별 구독 정보, 수정/삭제, 사용빈도 조절
3. **추가/편집** — 구독 입력 폼
4. **인사이트** — 카테고리 차트, AI 추천·해지 제안 (v1.2)
5. **설정** — 통화, 테마, 알림, 개인정보/AI 동의

## 10. 성공 기준 (스스로 점검용)

- MVP: 시뮬레이터에서 구독을 추가하고, 총액이 계산되고, 알림이 울린다.
- v1.2: AI가 최소 1개의 의미 있는 해지/추천 제안을 사유와 함께 보여준다.
- 출시: TestFlight로 본인·지인이 일주일간 실사용해도 크래시 없이 돈다.

## 11. 리스크 & 메모

- **범위 폭주 주의**: 결제 연동·가계부 기능까지 욕심내면 영영 못 끝냄. MVP 사수.
- **AI 비용**: LLM API는 호출당 과금. 무료 사용량·캐싱·호출 빈도 제한을 설계에 반영.
- **경쟁 앱 존재**: 구독 관리는 레드오션. 차별점은 "AI 기반 취향 추천·해지 제안"이므로 거기에 집중.
- **이름/상표**: 정한 앱 이름이 App Store·상표에 이미 있는지 확인 필요.

## 12. 지금 당장 할 일 (다음 액션)

1. Xcode 설치 및 새 프로젝트 생성 → (별첨) Xcode 실무 가이드 참고
2. `Subscription` SwiftData 모델 1개 정의
3. 구독을 추가하고 목록에 보여주는 화면 1개 완성
4. 그 위에 총액 계산 → 알림 → 차트 → AI 순으로 한 겹씩

> 한 번에 한 겹씩. MVP가 시뮬레이터에서 돌면 절반은 성공입니다.
